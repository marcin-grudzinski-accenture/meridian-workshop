Overview

The Meridian Components Inventory Dashboard is a single-page web application serving three warehouse locations (San Francisco, London, Tokyo). It gives operations staff real-time visibility into stock levels, order status, demand forecasts, financial performance, and purchase order recommendations.

The application is a two-tier system: a Vue 3 single-page application running in the browser and a FastAPI backend serving a REST/JSON API. All data is served from in-memory JSON files loaded at startup; there is no database in the current deployment.

API Endpoints

All endpoints are GET and return JSON. Base URL: http://localhost:8001/api. Interactive docs at http://localhost:8001/docs.

Inventory

GET/api/inventory
Query params: warehouse (opt), category (opt)
Response: Array of { id, sku, name, category, warehouse, quantity_on_hand, reorder_point, unit_cost, location, last_updated }
GET/api/inventory/{item_id}
Response: Single InventoryItem. 404 if not found.

Orders

GET/api/orders
Query params: warehouse, category, status (Delivered | Shipped | Processing | Backordered), month (YYYY-MM or Q#-YYYY)
Response: Array of { id, order_number, customer, items[], status, order_date, expected_delivery, total_value, actual_delivery?, warehouse?, category? }
GET/api/orders/{order_id}
Response: Single Order. 404 if not found.

Dashboard

GET/api/dashboard/summary
Query params: warehouse, category, status, month
Response: { total_inventory_value, low_stock_items, pending_orders, total_backlog_items, total_orders_value }

Demand & Backlog

GET/api/demand
Response: Array of { id, item_sku, item_name, current_demand, forecasted_demand, trend, period }. Trend values: increasing | stable | decreasing.
GET/api/backlog
Response: Array of { id, order_id, item_sku, item_name, quantity_needed, quantity_available, days_delayed, priority, has_purchase_order }. has_purchase_order is computed at runtime.

Restocking Recommendations

GET/api/restock
Query params: warehouse, category, budget (float, optional — applies ceiling)
Response: Array of { sku, name, category, warehouse, quantity_on_hand, reorder_point, forecasted_demand?, trend?, suggested_qty, unit_cost, estimated_cost }
Algorithm: Filters to items where quantity_on_hand ≤ reorder_point. Joins with demand forecasts by SKU. Sorts by urgency (increasing trend first, then by depletion ratio). If budget is supplied, applies a greedy ceiling — includes items in priority order until budget is exhausted.

Reports

GET/api/reports/quarterly
Query params: warehouse, category, status, month
Response: Array of { quarter, total_orders, total_revenue, delivered_orders, avg_order_value, fulfillment_rate }
GET/api/reports/monthly-trends
Query params: warehouse, category, status, month
Response: Array of { month (YYYY-MM), order_count, revenue, delivered_count }

Spending & Finance

GET/api/spending/summary
Response: { total_procurement_cost, total_operational_cost, total_labor_cost, total_overhead, *_change (% YoY) }
GET/api/spending/monthly
Response: Array of { month, procurement, operational, labor, overhead }
GET/api/spending/categories
Response: Array of { name, amount, percentage, color }
GET/api/spending/transactions
Response: Array of { id, date, description, category, warehouse, amount, vendor, type }

Data Layer

All data lives in server/data/ as JSON files. They are loaded into memory by server/mock_data.py at server startup. Changes to JSON files take effect on next server restart. There is no write path in the current implementation — all endpoints are read-only.

Composables client/src/composables/

useFilters.js

Singleton — all components share the same reactive filter state. Changes to any filter automatically trigger re-fetch in every view that watches it.

useI18n.js

Locale is persisted in localStorage under key app-locale. Currency is derived from locale (en → USD, ja → JPY).

useAuth.js

Mock authentication — no real auth server. Returns a hardcoded user object with locale-aware task list. Provides currentUser, isAuthenticated, logout(), getInitials(name).

Data Flow

The example below traces a filter interaction end-to-end: the operator selects Tokyo in the Location dropdown on the Orders page.

1
User interaction: Operator selects "Tokyo" in FilterBar's Location dropdown. FilterBar is bound directly to selectedLocation ref in useFilters.
2
Reactive update: selectedLocation.value changes to 'Tokyo'. The singleton composable propagates this to all listeners immediately.
3
Watch fires in Orders.vue: watch([selectedPeriod, selectedLocation, selectedCategory, selectedStatus], loadOrders) triggers. loadOrders() calls getCurrentFilters() which returns { warehouse: 'Tokyo', category: 'all', status: 'all' }.
4
API call via api.js: api.getOrders({ warehouse: 'Tokyo' }) builds URLSearchParams — only non-'all' values are appended. Axios sends GET http://localhost:8001/api/orders?warehouse=Tokyo.
5
FastAPI receives request: Route handler extracts warehouse='Tokyo'. Calls apply_filters(orders, warehouse='Tokyo'): [item for item in orders if item.get('warehouse') == 'Tokyo']. Then calls filter_by_month(filtered, None) — no-op. Returns Pydantic-validated list.
6
Response serialised: FastAPI serialises to JSON array. CORS middleware adds Access-Control-Allow-Origin: * header. HTTP 200 returned.
7
Frontend re-renders: Axios resolves; orders.value ref is updated with filtered array. Vue's reactivity engine diffs the DOM and re-renders only the changed table rows. If locale is Japanese, warehouse names and customer names are translated inline.

Deployment Topology

ComponentHostPortNotes
Frontend (Vite dev)localhost3000HMR enabled. Not for production — run npm run build and serve client/dist/ via nginx or similar.
Backend (Uvicorn)0.0.0.08001Binds all interfaces. API docs at /docs. CORS allows all origins — restrict before production deployment.
API base URLHardcoded in client/src/api.js line 3: http://localhost:8001/api. Update this for any non-local deployment.

Production considerations

  • Replace in-memory JSON with a real database (PostgreSQL recommended — schema maps cleanly from Pydantic models)
  • Restrict allow_origins in CORS middleware to the production frontend domain
  • Add authentication (the current application has none)
  • Update API_BASE_URL in api.js to the production backend URL or use an environment variable
  • Run frontend build (npm run build) and serve client/dist/ as static files

Running Locally

Prerequisites

  • Python 3.11+ with uv or a virtualenv with FastAPI, Uvicorn, Pydantic installed
  • Node.js 18+ and npm

Start the backend

cd server
uv run python main.py
# or: python main.py (with venv activated)
# → http://localhost:8001
# → API docs: http://localhost:8001/docs

Start the frontend

cd client
npm install   # first time only
npm run dev
# → http://localhost:3000

Run tests

# Backend API tests (51 tests)
cd tests
uv run pytest -v

# E2E browser tests (29 tests) — both servers must be running
cd ../   # project root
npm run test:e2e
